la correcta ejecución del proyecto TENNISFLOW, asegurando la integración de

┌──────────────────────────────────────────────────────────────────────────┐
│  TEMAS TRATADOS                                                          │
└──────────────────────────────────────────────────────────────────────────┘

1. Presentación general del proyecto y su alcance funcional
   
   - Problemática identificada: Manipulación manual de turnos en Club Puerto 
     Peñaliza (30 auxiliares + 8 boleadores, 12 canchas, 2 jornadas)
   
   - Solución propuesta: Sistema automatizado con registro timestamp, 
     algoritmo transparente y orden justo basado en datos reales
   
   - Casos de uso principales: Coordinador de Cancha (control total), 
     Profesor de Tenis (supervisión), Auxiliares (consulta)

2. Revisión del diseño de la base de datos y su integración con el sistema 
   de gestión de turnos
   
   - Modelado de entidades: Usuarios, Roles, Clubs, Canchas, Turnos, 
     Jornadas, Socios, Auxiliares
   
   - Relaciones críticas: Club-Canchas (1:N), Jornadas-Turnos (1:N), 
     Usuario-Roles (N:M)
   
   - Motor seleccionado: PostgreSQL por soporte robusto a transacciones y 
     escalabilidad
   
   - Migraciones: TypeORM para versionamiento automático del esquema

3. Aplicación de la metodología Kanban para la organización de tareas
   
   - Tablero Kanban con columnas: Backlog, To Do, In Progress, Review, Done
   
   - Límites WIP (Work In Progress): Máximo 3 tareas por desarrollador en 
     "In Progress"
   
   - Daily standups: 30 minutos diarios a las 9:00 AM para sincronización
   
   - Herramienta: GitHub Projects integrado con repositorio del código

4. Definición de roles y responsabilidades de cada integrante del equipo
   
   - Jhoan Gongora: Arquitectura general, módulos críticos (Auth, Turnos), 
     integración OAuth, despliegue final
   
   - Luisa Loaiza: Diseño UI/UX, componentes React, maquetación responsive, 
     accesibilidad WCAG 2.1
   
   - Juan Camilo Ávila: Modelado de datos, optimización de queries, 
     migraciones, documentación técnica

┌──────────────────────────────────────────────────────────────────────────┐
│  DECISIONES TOMADAS                                                      │
└──────────────────────────────────────────────────────────────────────────┘

✓ Stack tecnológico aprobado:
  - Backend: NestJS (Node.js + TypeScript) con arquitectura de microservicios
  - Frontend: React + Vite + TypeScript + TailwindCSS
  - Base de datos: PostgreSQL con TypeORM
  - Autenticación: JWT + OAuth 2.0 (Google)
  - Contenedores: Docker + Docker Compose

✓ Metodología de trabajo:
  - Kanban con GitHub Projects
  - Sprints de 2 semanas con entregas incrementales
  - Daily standups de 30 minutos
  - Code review obligatorio antes de merge a master

✓ Estándares de código:
  - ESLint + Prettier para formateo automático
  - Convenciones de nombres: camelCase (variables), PascalCase (componentes)
  - Commits descriptivos siguiendo Conventional Commits
  - Cobertura de pruebas mínima: 70% en backend, 60% en frontend

✓ Comunicación:
  - Canal principal: Microsoft Teams
  - Reuniones semanales: Lunes 14:00 PM (planificación) y Viernes 16:00 PM 
    (retrospectiva)
  - Documentación colaborativa: Google Docs + GitHub Wiki
